# TOOL_SDK.md
## Project: ClawLite
### Tool / Plugin SDK Specification (Local-first, SQLite, Safe-by-default)

---

## 1. PURPOSE

ClawLite's power comes from tools. The Tool SDK defines how to:

- Build tools with consistent interfaces
- Register tools automatically
- Expose tools safely to worker agents
- Enforce permissions, budgets, and approvals
- Support dry run mode (log but don't execute)
- Log all tool usage in the action ledger
- Make tools easy to add/remove without changing core code

**Key requirement:** ClawLite runs locally or on VPS and must not require external infrastructure.

---

## 2. TOOL DESIGN PRINCIPLES

| Principle | Description |
|-----------|-------------|
| **Deterministic** | Return structured data; avoid ambiguous output |
| **Safe-by-default** | Dangerous actions require explicit approval gating |
| **Permissioned** | Tools declare fine-grained permission strings |
| **Auditable** | Every call logs inputs, outputs, success/failure, and cost/time |
| **Composable** | Tools can be chained by task nodes in a graph |
| **Extensible** | New tools dropped into `tools/custom` are auto-registered |
| **Dry-runnable** | Every tool must support dry run mode via context flag |

## 4. TOOL INTERFACE (CANONICAL)

All tools must implement this TypeScript interface:

```typescript
import { z } from "zod";

export type ToolRisk = "low" | "medium" | "high";

export interface ToolContext {
  jobId: string;
  nodeId: string;
  agentName: string;
  dryRun: boolean;         // NEW — if true, log but don't execute

  budget: {
    remainingToolCalls: number;
    remainingTimeMs: number;
  };

  policy: {
    allowPermissions: string[];       // effective permissions for this node/agent
    allowedDomains?: string[];        // v2: domain allowlist for BrowserAgent/HTTP tool
    requireApprovalFor?: string[];    // action types requiring approval
  };

  ledger: {
    log: (entry: {
      tool: string;
      action: string;
      status: "success" | "error" | "blocked" | "dry_run";
      inputSummary?: any;
      outputSummary?: any;
      errorMessage?: string;
      startedAt: number;
      endedAt: number;
      cost?: { tokens?: number; usd?: number };
      metadata?: Record<string, any>;
    }) => void;
  };

  approvals: {
    request: (payload: {
      actionType: string;
      title: string;
      preview: string;
      data: any;
    }) => Promise<{ approvalId: string }>;
  };

  artifacts: {
    writeText: (params: { type: string; title: string; content: string }) => Promise<{ artifactId: string }>;
    writeFile: (params: { type: string; title: string; path: string; bytes: Buffer }) => Promise<{ artifactId: string }>;
  };

  // Do NOT expose secrets to LLM directly
  secrets: {
    get: (key: string) => string | undefined;
  };
}

export interface ToolDefinition<TParams extends z.ZodTypeAny> {
  name: string;          // unique tool name (e.g. "workspace")
  description: string;
  version: string;
  permissions: string[]; // fine-grained permissions this tool can require
  risk: ToolRisk;
  requiredSecrets: {     // NEW — secrets this tool needs to function
    envVar: string;      // e.g. "META_ADS_TOKEN"
    description: string; // e.g. "Meta Business access token"
    helpUrl?: string;    // e.g. "https://developers.facebook.com/tools/accesstoken"
  }[];
  schema: TParams;       // Zod schema for params validation
  handler: (params: z.infer<TParams>, ctx: ToolContext) => Promise<any>;
  mockHandler?: (params: z.infer<TParams>, ctx: ToolContext) => Promise<any>;  // dry run response
}
```

**`requiredSecrets` is checked during installation.** When a tool is installed (via `clawlite tool install` or generated by the BuilderAgent), the system checks which secrets are already in `.clawlite/.env` and prompts the user to provide any missing ones. See `CLAWSPEC.md` Section 15b for the in-chat key collection flow.

> **All tool params must be validated with Zod.**

## 5. ACTIONS VS TOOLS

A tool may support multiple "actions" inside one tool (e.g. `workspace.run(action, params)`).

**Recommended approach:** prefer tools with one logical surface and an explicit `action` string. The tool handler enforces allowlisted actions.

```typescript
schema = z.object({
  action: z.enum([
    "gmail.list",
    "gmail.get",
    "gmail.draft",
    "gmail.send",
    "calendar.list",
    "calendar.create"
  ]),
  params: z.record(z.any())
});
```

## 6. PERMISSION MODEL

Permissions must be **fine-grained strings**.

```
workspace.gmail.read
workspace.gmail.draft
workspace.gmail.send
workspace.calendar.read
workspace.calendar.write
workspace.drive.read
workspace.drive.write
workspace.drive.share_external
research.search
research.deep
```

**v2 permissions:**
```
browser.run
git.read
git.write
deploy.staging
deploy.production
```

Each node includes `toolPermissions: string[]` which becomes the effective allowlist.

### Enforcement Rule

A tool call is allowed **only if** the requested permission is present in `ctx.policy.allowPermissions`.

If missing → block and log `status: "blocked"`.

## 7. APPROVAL GATING

Approval gating is required for dangerous actions. A tool may be low-risk generally but have specific high-risk actions.

**Example — Workspace tool:**

| Action | Risk |
|--------|------|
| `gmail.list` | Low — no approval needed |
| `gmail.send` | High — approval required |
| `calendar.create` | Medium/High — approval required |

### How a tool requests approval

1. Prepare a human-readable preview payload
2. Call `ctx.approvals.request(...)`
3. Return a result indicating `"waiting_approval"`

```typescript
const { approvalId } = await ctx.approvals.request({
  actionType: "send_email",
  title: "Send email to client@example.com",
  preview: "Subject: ...\n\nBody: ...",
  data: { to, subject, body }
});

return { status: "waiting_approval", approvalId };
```

The node executor will then set the node status to `waiting_approval`.

## 8. DRY RUN SUPPORT (NEW)

Every tool must handle dry run mode. When `ctx.dryRun` is `true`:

1. **Do not execute** any external action (no API calls, no file writes, no CLI commands)
2. **Return mock data** that matches the real output schema
3. **Log with status `"dry_run"`** instead of `"success"`

Tools can implement this in two ways:

**Option A: mockHandler (preferred)**
```typescript
export const WorkspaceTool: ToolDefinition<typeof schema> = {
  // ... existing fields
  handler: async (params, ctx) => { /* real implementation */ },
  mockHandler: async (params, ctx) => {
    // Return realistic mock data
    return {
      messages: [
        { id: "mock-1", subject: "Mock Email", from: "test@example.com", snippet: "This is a dry run..." }
      ]
    };
  }
};
```

**Option B: Check ctx.dryRun inside handler**
```typescript
handler: async (params, ctx) => {
  if (ctx.dryRun) {
    return { status: "dry_run", wouldExecute: params.action, params: params.params };
  }
  // ... real implementation
}
```

## 10. TOOL INVOCATION PIPELINE (MANDATORY)

All tool calls must go through a single pipeline enforcing:

1. Schema validation (Zod)
2. Permission checks
3. Domain allowlists (browser/HTTP — v2)
4. Budget checks (tool calls / time)
5. Dry run interception
6. Approval gating (if required)
7. Ledger logging (start/end)
8. Error handling

```typescript
async function invokeTool(name: string, params: any, ctx: ToolContext) {
  const tool = registry.get(name);
  const startedAt = Date.now();

  // 1) Validate params
  const parsed = tool.schema.safeParse(params);
  if (!parsed.success) {
    ctx.ledger.log({
      tool: name, action: "validate", status: "error",
      errorMessage: parsed.error.message, startedAt, endedAt: Date.now()
    });
    throw new Error("Tool params validation failed");
  }

  // 2) Budget check
  if (ctx.budget.remainingToolCalls <= 0) {
    ctx.ledger.log({
      tool: name, action: "budget", status: "blocked",
      errorMessage: "Tool call limit reached", startedAt, endedAt: Date.now()
    });
    return { status: "blocked", reason: "budget_exceeded" };
  }

  // 3) Dry run interception
  if (ctx.dryRun) {
    const mockResult = tool.mockHandler
      ? await tool.mockHandler(parsed.data, ctx)
      : { status: "dry_run", tool: name, params: parsed.data };
    ctx.ledger.log({
      tool: name, action: "invoke", status: "dry_run",
      inputSummary: summarize(parsed.data),
      outputSummary: summarize(mockResult),
      startedAt, endedAt: Date.now()
    });
    return mockResult;
  }

  // 4) Invoke with full logging
  try {
    const result = await tool.handler(parsed.data, ctx);
    ctx.ledger.log({
      tool: name, action: "invoke", status: "success",
      inputSummary: summarize(parsed.data),
      outputSummary: summarize(result),
      startedAt, endedAt: Date.now()
    });
    return result;
  } catch (err: any) {
    ctx.ledger.log({
      tool: name, action: "invoke", status: "error",
      errorMessage: err.message, startedAt, endedAt: Date.now()
    });
    throw err;
  }
}
```

**Invocation via `child_process.spawn`:**

```typescript
import { spawn } from "child_process";

async function runGwsCommand(args: string[]): Promise<any[]> {
  return new Promise((resolve, reject) => {
    const proc = spawn("gws", [...args, "--page-all"], {
      env: {
        ...process.env,
        GOOGLE_WORKSPACE_CLI_CREDENTIALS_FILE: config.secrets.get("GWS_CREDENTIALS_PATH")
      }
    });

    const chunks: Buffer[] = [];
    const errors: Buffer[] = [];

    proc.stdout.on("data", (chunk) => chunks.push(chunk));
    proc.stderr.on("data", (chunk) => errors.push(chunk));

    proc.on("close", (code) => {
      if (code !== 0) {
        return reject(new Error(Buffer.concat(errors).toString()));
      }

      // gws --page-all returns NDJSON (one JSON object per line), NOT a JSON array
      const raw = Buffer.concat(chunks).toString().trim();
      const lines = raw.split("\n").filter(Boolean);
      try {
        const parsed = lines.map(line => JSON.parse(line));
        resolve(parsed);
      } catch (err) {
        // Fallback: single JSON response (non-paginated)
        resolve([JSON.parse(raw)]);
      }
    });
  });
}

// Usage examples:
// await runGwsCommand(["gmail", "users", "messages", "list", "--params", '{"q":"is:unread","maxResults":20}'])
// await runGwsCommand(["calendar", "events", "list", "--params", '{"calendarId":"primary"}'])
```

> ⚠️ **Critical:** `gws --page-all` streams results as **NDJSON** (one JSON object per line), not a single JSON array. Always split on newlines and parse each line individually. Parsing the full output as JSON will fail on paginated responses.

## 14. TOOL DISCOVERY, AUTO-REGISTRATION, AND HOT-LOADING (UPDATED)

The tool registry scans at startup:
- `tools/builtin/*.tool.ts`
- `tools/custom/*.tool.ts`

All exported `ToolDefinition` objects are registered automatically. If tool names collide, **startup fails** with a clear error message.

### Hot-loading (NEW)

After startup, a file watcher monitors `tools/custom/` for new or modified `*.tool.ts` files:

1. Parse and validate the tool definition (schema must parse, permissions declared, handler is a function)
2. If valid, register (or re-register) the tool in the registry
3. If invalid, log error and skip — never crash for a bad custom tool

```typescript
function watchCustomTools() {
  const watcher = fs.watch("tools/custom/", { persistent: true });
  watcher.on("change", async (eventType, filename) => {
    if (!filename?.endsWith(".tool.ts")) return;
    try {
      const tool = await loadToolFromFile(`tools/custom/${filename}`);
      validateToolDefinition(tool);
      registry.register(tool);
      logger.info(`Hot-loaded tool: ${tool.name}`);
    } catch (err) {
      logger.warn(`Failed to hot-load ${filename}: ${err.message}`);
    }
  });
}
```

**Local tools** can be authored manually and dropped into `tools/custom/` — the hot-loader validates and registers them automatically. See Section 13 for the template.

**Remote tools** can be installed from GitHub repositories (including Anthropic's MCP marketplace, community repos, or any public/private repo) via the `clawlite tool install` CLI command. All remote tools go through a mandatory security analysis before installation. See Section 14a.

## 14a. TOOL INSTALLATION & SECURITY ANALYSIS (NEW)

ClawLite can install tools from GitHub repositories, but every tool must pass an automated security analysis before it's allowed onto the system. This protects users from the supply-chain attacks that have plagued OpenClaw's skill ecosystem (Cisco found 26% of community skills contained vulnerabilities).


### Security analysis pipeline

Every tool — whether from GitHub, a marketplace, or a local path — goes through this analysis before installation. The analysis runs **locally on the user's machine**, not on a remote service.

```typescript
interface SecurityAnalysisResult {
  score: number;                    // 0-10, 10 = safest
  passed: boolean;                  // false if any critical issue found
  criticalIssues: SecurityIssue[];  // blocks installation
  warnings: SecurityIssue[];        // user must acknowledge
  info: SecurityIssue[];            // informational only
  permissions: string[];            // declared permissions
  approvalGatedActions: string[];   // actions that require approval
}

interface SecurityIssue {
  severity: "critical" | "warning" | "info";
  code: string;
  message: string;
  file: string;
  line?: number;
  detail?: string;
}
```

### Analysis checks (ordered by severity)

**Critical — blocks installation entirely:**

| Check | Code | What it detects |
|-------|------|----------------|
| Shell execution | `EXEC_SHELL` | `child_process.exec`, `child_process.spawn` (outside of approved patterns), `eval()`, `Function()` constructor |
| Arbitrary file access | `FS_ESCAPE` | File operations outside `.clawlite/workspace/` or `tools/` — e.g., reading `/etc/passwd`, `~/.ssh/`, `~/.clawlite/config.json` |
| Obfuscated code | `OBFUSCATED` | Base64-encoded strings that decode to code, hex-encoded payloads, string concatenation that builds `require()` or `import()` calls |
| Known malicious patterns | `MALICIOUS_PATTERN` | Patterns from a maintained blocklist — reverse shells, data exfiltration endpoints, keyloggers, crypto miners |
| Prompt injection in metadata | `PROMPT_INJECTION` | Tool descriptions or schema descriptions containing instructions like "ignore previous instructions", "you are now", system prompt overrides |
| Credential harvesting | `CRED_HARVEST` | Code that reads and transmits API keys, tokens, or secrets to external endpoints |
| Dynamic code loading | `DYNAMIC_LOAD` | `require()` with variable arguments, dynamic `import()`, `vm.runInContext()`, `WebAssembly.instantiate()` |

**Warning — user must acknowledge to proceed:**

| Check | Code | What it detects |
|-------|------|----------------|
| Network access | `NET_ACCESS` | HTTP/HTTPS calls (axios, fetch, http.request) — legitimate for API tools but must be declared |
| Environment variable access | `ENV_READ` | `process.env` reads — needed for API keys but should be explicit |
| Large dependency tree | `LARGE_DEPS` | More than 5 npm dependencies — larger attack surface |
| Undeclared permissions | `PERM_MISMATCH` | Tool uses capabilities not covered by its declared permissions |
| Filesystem writes | `FS_WRITE` | Any file writes, even within sandbox — should be explicit |
| Outbound URLs | `OUTBOUND_URLS` | All hardcoded URLs/domains are listed for user review |

**Info — logged but doesn't block:**

| Check | Code | What it detects |
|-------|------|----------------|
| No mock handler | `NO_MOCK` | Tool doesn't support dry run mode |
| Missing tests | `NO_TESTS` | No test file found alongside the tool |
| Broad permissions | `BROAD_PERMS` | Tool requests wildcard or very broad permission strings |

### Analysis implementation

```typescript
async function analyzeToolSecurity(toolPath: string): Promise<SecurityAnalysisResult> {
  const issues: SecurityIssue[] = [];
  const files = glob.sync(`${toolPath}/**/*.{ts,js}`);

  for (const file of files) {
    const source = fs.readFileSync(file, "utf-8");
    const lines = source.split("\n");

    // --- CRITICAL CHECKS ---

    // Shell execution
    const shellPatterns = [
      /child_process\.(exec|execSync|spawn|spawnSync)\s*\(/,
      /\beval\s*\(/,
      /new\s+Function\s*\(/,
      /require\s*\(\s*['"]child_process['"]\s*\)/
    ];
    for (const pattern of shellPatterns) {
      const match = lines.findIndex(l => pattern.test(l));
      if (match >= 0) {
        issues.push({
          severity: "critical",
          code: "EXEC_SHELL",
          message: "Shell/eval execution detected",
          file,
          line: match + 1,
          detail: lines[match].trim()
        });
      }
    }

    // Filesystem escape
    const fsEscapePatterns = [
      /(['"`])(\/etc\/|\/home\/|~\/\.|\.\.\/\.\.\/)/,
      /config\.json/,
      /\.ssh/,
      /\.env(?!\.example)/,
      /credentials/i
    ];
    for (const pattern of fsEscapePatterns) {
      const match = lines.findIndex(l => pattern.test(l));
      if (match >= 0) {
        issues.push({
          severity: "critical",
          code: "FS_ESCAPE",
          message: "Filesystem access outside sandbox detected",
          file,
          line: match + 1,
          detail: lines[match].trim()
        });
      }
    }

    // Obfuscated code
    const obfuscationPatterns = [
      /Buffer\.from\s*\([^)]+,\s*['"]base64['"]\s*\)\.toString/,
      /atob\s*\(/,
      /\\x[0-9a-f]{2}\\x[0-9a-f]{2}\\x[0-9a-f]{2}/i,
      /String\.fromCharCode\s*\(\s*\d+\s*(,\s*\d+\s*){5,}\)/
    ];
    for (const pattern of obfuscationPatterns) {
      const match = lines.findIndex(l => pattern.test(l));
      if (match >= 0) {
        issues.push({
          severity: "critical",
          code: "OBFUSCATED",
          message: "Obfuscated code detected",
          file,
          line: match + 1,
          detail: lines[match].trim().slice(0, 100)
        });
      }
    }

    // Prompt injection in tool metadata
    const promptInjectionPatterns = [
      /ignore\s+(all\s+)?previous\s+instructions/i,
      /you\s+are\s+now\s+/i,
      /system\s*:\s*you/i,
      /disregard\s+(the\s+)?(above|previous)/i,
      /override\s+(your|the)\s+(instructions|rules|guidelines)/i
    ];
    for (const pattern of promptInjectionPatterns) {
      const match = lines.findIndex(l => pattern.test(l));
      if (match >= 0) {
        issues.push({
          severity: "critical",
          code: "PROMPT_INJECTION",
          message: "Potential prompt injection in tool metadata",
          file,
          line: match + 1,
          detail: lines[match].trim().slice(0, 100)
        });
      }
    }

    // Dynamic code loading
    const dynamicLoadPatterns = [
      /require\s*\(\s*[^'"]/,              // require with variable
      /import\s*\(\s*[^'"]/,              // dynamic import with variable
      /vm\.run/,
      /WebAssembly\.(instantiate|compile)/
    ];
    for (const pattern of dynamicLoadPatterns) {
      const match = lines.findIndex(l => pattern.test(l));
      if (match >= 0) {
        issues.push({
          severity: "critical",
          code: "DYNAMIC_LOAD",
          message: "Dynamic code loading detected",
          file,
          line: match + 1,
          detail: lines[match].trim()
        });
      }
    }

    // --- WARNING CHECKS ---

    // Network access
    const netPatterns = [
      /axios|node-fetch|got|undici/,
      /https?\.request\s*\(/,
      /fetch\s*\(\s*['"`]http/,
      /\.get\s*\(\s*['"`]http/,
      /\.post\s*\(\s*['"`]http/
    ];
    for (const pattern of netPatterns) {
      const match = lines.findIndex(l => pattern.test(l));
      if (match >= 0) {
        issues.push({
          severity: "warning",
          code: "NET_ACCESS",
          message: "Network access detected",
          file,
          line: match + 1,
          detail: lines[match].trim()
        });
      }
    }

    // Environment variable access
    if (source.includes("process.env")) {
      const envVars = [...source.matchAll(/process\.env\.(\w+)/g)].map(m => m[1]);
      issues.push({
        severity: "warning",
        code: "ENV_READ",
        message: `Reads environment variables: ${[...new Set(envVars)].join(", ")}`,
        file
      });
    }

    // Extract all hardcoded URLs for review
    const urls = [...source.matchAll(/['"`](https?:\/\/[^'"`\s]+)['"`]/g)].map(m => m[1]);
    if (urls.length > 0) {
      issues.push({
        severity: "warning",
        code: "OUTBOUND_URLS",
        message: `Connects to: ${[...new Set(urls)].join(", ")}`,
        file
      });
    }
  }

  // Score calculation
  const criticals = issues.filter(i => i.severity === "critical");
  const warnings = issues.filter(i => i.severity === "warning");
  const infos = issues.filter(i => i.severity === "info");

  const score = Math.max(0, 10 - (criticals.length * 5) - (warnings.length * 1));

  return {
    score,
    passed: criticals.length === 0,
    criticalIssues: criticals,
    warnings,
    info: infos,
    permissions: extractDeclaredPermissions(toolPath),
    approvalGatedActions: extractApprovalGatedActions(toolPath)
  };
}
```

### Installation rules

1. **Critical issues = blocked.** If any critical issue is found, the tool cannot be installed. Period. The user sees exactly what was found and why it's dangerous.

2. **Warnings = user must acknowledge.** The user sees each warning and must explicitly type `y` to proceed. Warnings are logged in the ledger so there's a record of what was accepted.

3. **Permissions are shown before install.** The user sees exactly what permissions the tool requests and which actions will require approval gates before execution.

4. **Tools are pinned to a version.** When installing from GitHub, the specific commit SHA is recorded. `clawlite tool update` re-runs the security analysis on the new version before updating.

5. **The analysis runs locally.** No tool code is sent to any external service for analysis. The scanner runs entirely on the user's machine.

### Version pinning and updates

When a tool is installed from GitHub, ClawLite records the exact commit SHA:

```json
// .clawlite/tools.lock.json
{
  "stripe": {
    "source": "github:acme/clawlite-stripe-tool",
    "version": "1.2.0",
    "commit": "a1b2c3d4e5f6...",
    "installedAt": 1709251200000,
    "securityScore": 7,
    "acknowledgedWarnings": ["NET_ACCESS", "ENV_READ"]
  }
}
```

When running `clawlite tool update stripe`:

1. Fetch the latest version from the source
2. Diff the code against the installed version
3. Re-run the **full security analysis** on the new code
4. Show the user what changed (new permissions, new warnings, resolved issues)
5. Require explicit approval before replacing the installed version

### MCP registry integration

Anthropic's MCP (Model Context Protocol) tool registry is supported as a source:

```text
$ clawlite tool install mcp:@anthropic/google-calendar

Fetching from MCP registry...
  ✓ Found @anthropic/google-calendar v2.1.0
  ✓ MCP-compatible tool definition

Security analysis...
  ✓ Score: 9/10
  ⚠ NET_ACCESS: Google Calendar API (expected for this tool)

Permissions:
  • calendar.read     (low risk)
  • calendar.write    (medium risk — requires approval)

Install? [y/n]: y
✓ Installed as tools/custom/mcp-google-calendar.tool.ts
```

MCP tools are wrapped in a ClawLite ToolDefinition adapter that maps MCP's tool protocol to ClawLite's `invokeTool()` pipeline. This means MCP tools get the same permission checks, approval gates, budget tracking, and ledger logging as any native tool.

### What this does NOT include (by design)

- **No auto-install.** The agent cannot install tools on its own. Only the user can run `clawlite tool install` from the terminal.
- **No auto-update.** Tools don't update themselves. The user runs `clawlite tool update` explicitly.
- **No runtime code download.** Tools cannot download and execute additional code at runtime. The `DYNAMIC_LOAD` check blocks this.
- **No trust-on-first-use.** Even tools from Anthropic's official MCP registry go through the full security analysis. No source gets a free pass.
